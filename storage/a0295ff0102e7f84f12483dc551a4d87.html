Why this was a good test for Colab MCP

Most product demos stay in safe territory. Load a CSV. Draw a chart. Maybe install one package.

I wanted something messier.

Quantum chemistry is a good stress test because it combines several failure points at once:

  • dependency management
  • scientific Python tooling
  • cloud execution
  • notebook organization
  • runtime debugging
  • optional handoff to real hardware

If Colab MCP could survive that stack, it would tell me much more about its real-world usefulness than a toy notebook ever could.

The setup: Colab MCP

Google's new Colab MCP server gives an MCP-compatible agent programmatic control over a Google Colab notebook. Not read-only access. Real notebook control.

That means an agent can:

  • create and edit cells
  • write markdown and code
  • execute cells
  • inspect outputs and tracebacks
  • iterate until the notebook actually works

The setup is simple. Add the MCP server to your agent config:

"mcpServers": {
  "colab-proxy-mcp": {
    "command": "uvx",
    "args": ["git+https://github.com/googlecolab/colab-mcp"],
    "timeout": 30000
  }
}

Prerequisites are minimal:

  • Python
  • git
  • uv

Once the server is configured, it connects to the Colab notebook you already have open in your browser.

The task: compute the H2 potential energy surface

I chose the hydrogen molecule, H₂.

This is the smallest meaningful benchmark in quantum chemistry. It is simple enough to run quickly, but still rich enough to show whether the quantum workflow is doing something real.

The goal was to compute how the molecule's energy changes as the bond between the two hydrogen atoms is stretched. This is called the potential energy surface.

Why does that matter?

Because it lets you compare three things directly:

  • a standard classical approximation
  • a quantum algorithm
  • the exact reference solution

That makes H₂ the perfect test case for both the scientific result and the tooling around it.

At equilibrium geometry, the agent ran three methods:

Hartree-Fock -1.11700 20.31 mHa
VQE (simulator) -1.13731 0.00 mHa
Exact diagonalization -1.13731 -

This part mattered scientifically.

The VQE result matched the exact answer on the simulator, while Hartree-Fock left a noticeable gap. That is the core promise of quantum chemistry workflows like this: quantum methods can recover effects that standard approximations miss.

But the first VQE attempt did not work.

The agent hit:

AlgorithmError: The primitive job to evaluate the energy failed!

The cause was a version mismatch between the primitive being used and the API style expected by qiskit-algorithms.

The fix was to switch to StatevectorEstimator, rerun the notebook, and continue.

That debugging loop is the real value of Colab MCP. The agent does not just generate code. It sees the traceback, changes the notebook, and re-executes until the workflow is alive again.

So this was not just an orchestration demo. The experiment also captured a real chemistry phenomenon that motivates quantum algorithms in the first place.

Then I pushed it to real quantum hardware

After the simulator run, I wanted to see whether the same workflow would survive contact with real hardware.

The agent connected to IBM Quantum using a token stored in Colab Secrets:

from google.colab import userdata
IBM_TOKEN = userdata.get('IBM_TOKEN')

service = QiskitRuntimeService(
    channel="ibm_quantum_platform",
    token=IBM_TOKEN
)

Available backends included:

  • ibm_fez: 156 qubits, 0 pending jobs
  • ibm_torino: 133 qubits, 0 pending jobs
  • ibm_kingston: 156 qubits, 119 pending jobs
  • ibm_marrakesh: 156 qubits, 0 pending jobs

The agent selected ibm_fez.

Then things got messy again.

More breakage, more fixes

The IBM runtime transpiler plugins were not compatible with the Qiskit 1.4.x environment used by the chemistry stack.

That caused:

TranspilerError: Invalid plugin name ibm_dynamic_circuits

The fix was to bypass the plugin path and transpile directly against the backend's basis gates and coupling map.

Then another issue appeared: IBM's open plan did not support Sessions in this setup.

So the agent switched from a Session-based flow to:

EstimatorV2(mode=backend)

Again, this is where the workflow becomes meaningful.

The agent was not just writing a notebook. It was navigating a real compatibility chain across Colab, Qiskit, and IBM Runtime.

That is not a failure. That is the current reality of NISQ-era quantum computing.

The algorithm worked. The notebook worked. The cloud workflow worked. But physical qubits still introduce noise through gate error, decoherence, and measurement error.

And that contrast was valuable to see in one end-to-end run.

Why this experiment was meaningful

There were two reasons this was worth doing.

1. It showed a real scientific difference

H₂ is small, but meaningful.

It is the smallest benchmark that still exposes whether the algorithmic stack is doing real work. The simulator showed that VQE could recover the exact solution for this system, and the full bond scan highlighted where classical approximation methods begin to fail.

2. It showed a new way to work with agents

This was the bigger surprise.

Colab MCP made it practical to hand a messy technical objective to an agent and let it operate inside a cloud notebook until the notebook actually worked.

That included:

  • package installation
  • notebook structure
  • traceback-driven debugging
  • reruns after fixes
  • cloud execution
  • optional handoff to real quantum hardware

That is much more interesting than simple code generation.

It points toward a new working style where Colab becomes an executable cloud workspace for AI agents, not just a place humans paste code into.

What I learned

Colab MCP is genuinely useful

This did not feel like a gimmick.

The agent created 17 cells, handled several compatibility issues, ran a 15-point scientific sweep, submitted a real quantum job, and produced usable outputs quickly.

That is exactly the kind of workflow where local-only agent setups start to feel cramped.

The package ecosystem is still the roughest part

The biggest friction was not the quantum algorithm.

It was Python package compatibility across Qiskit, chemistry libraries, and IBM runtime tooling.

That is where a notebook-controlling agent helps most. It can read the errors and iterate much faster than a human manually bouncing between terminal, browser, and documentation.

Real hardware is still noisy

The simulator matched the exact answer.

The hardware result came back with 1.21% relative error on a very small problem. That is a useful reminder that the software stack may be getting smoother while the hardware frontier is still about error mitigation and long-term fault tolerance.

Colab is an underrated chemistry environment

PySCF installs cleanly in Colab's Linux runtime and avoids the setup pain many people hit on Windows.

For this kind of workflow, the combination of Colab plus MCP is not just convenient. It may actually be the easiest path from idea to result.

How to reproduce it

  1. Set up the Colab MCP server in your agent config
  2. Open a Google Colab notebook in your browser
  3. Add your IBM Quantum token to Colab Secrets as IBM_TOKEN
  4. Give the agent a concrete task and let it build the notebook

In my case, the task was ambitious enough to break a few times, which is exactly why it was useful.

Final take

Google's Colab MCP announcement is not just about making notebook automation easier.

It changes what Colab can be.

Instead of treating Colab as a notebook you manually edit, you can start treating it as a cloud workspace an agent can actively operate.

This experiment pushed that idea through a workflow that included scientific Python, version conflicts, notebook authoring, parameter sweeps, and real hardware submission.

The result was not magic. It was a real notebook, built in the cloud, debugged in place, and executed end to end by an agent.

That is why I think Colab MCP is worth paying attention to.

> If this was my first stress test of Colab MCP, it was a very convincing one.
