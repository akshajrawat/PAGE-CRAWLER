Gemma-SRE: Self-Hosted vLLM Infrastructure Agent

#gemma #mcpserver #tpusprint #vllm

Gemma-SRE is a high-performance, private DevOps and Site Reliability Engineering (SRE) assistant designed for the TPU Sprint. It leverages Gemma 2 models, self-hosted via vLLM on Google Cloud Run GPUs (NVIDIA L4) or GKE TPUs (v5e), to bridge the gap between cloud infrastructure and intelligent automation.

What is this project trying to Do?

By integrating directly with Google Cloud Logging and the Vertex AI Model Registry, the SRE agent autonomously analyzes system errors, suggests production-ready remediation plans, and manages its own scalable inference stack. The project features a robust Model-as-Code approach, providing automated deployment templates that include optimized configurations for startup probes, GCS FUSE storage, and XLA-accelerate performance, ensuring a secure and cost-efficient private inference endpoint for any Google Cloud project.

Is this yet another Python MCP Demo?

Yes.

Python has traditionally been the main coding language for ML and AI tools. The goal of this article is to provide a minimal viable vLLM MCP server that can be used with tools such as Gemini CLI or the Agent Development Kit (ADK).

So — What does this Project Actually Do ?

This project is an automated SRE and DevOps assistant that uses a private, self-hosted AI (Gemma) to help you manage and troubleshoot Google Cloud infrastructure for deploying self-hosted models.

Specifically, it performs three main functions:

AI-Driven Troubleshooting

The agent connects to Google Cloud Logging to identify errors in your environment. It uses a self-hosted vLLM inference server to:

  • Summarize Logs: Turn raw, messy error logs into high-level summaries of what is actually failing.
  • Suggest Remediation: Provide expert 3-step fix plans for specific incidents (like memory limits being exceeded or container crashes).

Infrastructure Automation

The project includes a full "Inference Stack" manager. It can:

  • Deploy vLLM to Cloud Run: Automatically generate and execute commands to deploy an NVIDIA L4 GPU-accelerated server
  • Manage Scaling: Adjust min/max instances and check the health of the inference service
  • Generate Configs: Create Kubernetes (GKE) manifests for running vLLM on TPU v5e chips.

Model Weight Management

It bridges the gap between model providers and your private cloud:

  • Vertex AI & Kaggle Integration: Provides guides and commands to pull Gemma model weights from Vertex AI Model Garden or Kaggle
  • GCS FUSE Setup: Handles the complex configuration needed to mount a Google Cloud Storage bucket as a local disk for vLLM, allowing you to run models without manually baking them into container images.

In short: It is an MCP (Model Context Protocol) server that gives an LLM (like Claude or Gemini) the "hands" to see your logs and the "tools" to deploy and manage its own TPU and GPU-powered inference infrastructure.

Where do I start?

The strategy for starting MCP development for vLLM management is a incremental step by step approach.

First, the basic development environment is setup with the required system variables, and a working Gemini CLI configuration.

Then, a minimal Python MCP Server is built with stdio transport. This server is validated with Gemini CLI in the local environment.

This setup validates the connection from Gemini CLI to the local server via MCP. The MCP client (Gemini CLI) and the Python MCP server both run in the same local environment.

Setup the Basic Environment

At this point you should have a working Python environment and a working Gemini CLI installation. The next step is to clone the GitHub samples repository with support scripts:

cd ~
git clone https://github.com/xbill9/aisprintapr2026

Then run init.sh from the cloned directory.

The script will attempt to determine your shell environment and set the correct variables:

cd aisprintapr2026
source init.sh

If your session times out or you need to re-authenticate- you can run the set_env.sh script to reset your environment variables:

cd aisprintapr2026
source set_env.sh

Variables like PROJECT_ID need to be setup for use in the various build scripts- so the set_env script can be used to reset the environment if you time-out.

vLLM Management Tool with MCP Stdio Transport

One of the key features that the standard MCP libraries provide is abstracting various transport methods.

The high level MCP tool implementation is the same no matter what low level transport channel/method that the MCP Client uses to connect to a MCP Server.

The simplest transport that the SDK supports is the stdio (stdio/stdout) transport — which connects a locally running process. Both the MCP client and MCP Server must be running in the same environment.

The connection over stdio will look similar to this:

# Initialize FastMCP server
mcp = FastMCP("Self-Hosted vLLM DevOps Agent")

vLLM Lifecycle Management via MCP

The MCP tools provide a complete suite of agent-oriented operations for managing vLLM deployment on Cloud Run or a TPU.

Overview of Tools 

## 🛠 Available Tools

The following tools are available via the MCP server:

### Infrastructure Management
* **`deploy_vllm`** : Deploys a new vLLM instance on Cloud Run GPU.
* **`destroy_vllm`** : Deletes the Cloud Run vLLM service.
* **`status_vllm`** : Checks the health and status of the vLLM service.
* **`update_vllm_scaling`** : Adjusts the minimum and maximum instances for scaling.
* **`get_vllm_deployment_config`** : Generates `gcloud` commands for deployment.

### Model Management
* **`list_vertex_models`** : Lists models in the Vertex AI Registry.
* **`list_bucket_models`** : Lists model artifacts in your GCS bucket.
* **`get_vertex_ai_model_copy_instructions`** : Guide to transfer Gemma models from Vertex AI.
* **`get_kaggle_model_copy_instructions`** : Guide to transfer Gemma models from Kaggle.

### AI & Operations
* **`analyze_cloud_logging`** : Summarizes error logs using self-hosted vLLM.
* **`suggest_sre_remediation`** : Provides 3-step plans for SRE incidents.
* **`query_vllm`** : Sends direct prompts to the self-hosted Gemma model.
* **`get_vllm_tpu_deployment_config`** : Generates GKE manifests for TPU v5e.

mcp_vllm-devops-agent_suggest_sre_remediation

This tool reviews the logs and provides suggestions on fixing deployment issues.

✦ I have successfully used the suggest_sre_remediation tool to analyze the container startup failure (PORT=8080 timeout error).

  🩺 SRE Remediation Plan for Container Startup Failure

  1. Root Cause Identification
   * Log Correlation: Check vllm-gemma-2b-it logs for a mismatch between the container's listening port and the Cloud Run default port.
   * Port Check: Confirm if vLLM is using its default port (8000) while the Cloud Run environment expects port 8080.
   * Timeout Check: Verify if the startup-probe timeout is sufficient for the vLLM model (Gemma-2b-it) to load into GPU memory.

  2. Corrective Actions
   * Port Alignment: Update the Cloud Run service configuration to listen on port 8000 or explicitly set the --port argument in the vLLM
     command.
   * Timeout Extension: Increase the initialDelaySeconds and timeoutSeconds for the startupProbe to account for large model weight loading
     from GCS FUSE.
   * Scaling Check: Ensure that gpu-memory-utilization is not set too high (default 0.9), as this can cause OOM errors during initialization.

  3. Verification & Validation
   * Redeploy: Apply the corrected configuration using gcloud run services update.
   * Health Probe: Verify the /health endpoint returns 200 OK on port 8000.
   * Uptime Monitoring: Confirm the service stays Ready after the initial 180s delay.

  The service is currently healthy after the latest redeployment with the corrected port settings.

 > summarize the project
✦ This project is an MCP (Model Context Protocol) Server designed as an automated DevOps/SRE assistant. It leverages self-hosted Gemma models
  running on vLLM (Cloud Run GPU) to analyze Google Cloud infrastructure logs and provide actionable remediation strategies.

  Core Capabilities
   * Log Intelligence: Automatically fetches and summarizes Cloud Logging errors (severity=ERROR) to identify root causes using private LLM
     inference.
   * Incident Response: Generates expert 3-step remediation plans for specific SRE incidents (e.g., OOMKilled, connectivity issues).
   * Infrastructure Lifecycle: Provides tools to deploy, scale, and destroy vLLM inference stacks on Cloud Run using NVIDIA L4 GPUs and GCS
     FUSE.
   * Model Portability: Includes automated workflows to transfer Gemma weights from Vertex AI Model Garden or Kaggle to private GCS buckets
     for use with vLLM.

  Technical Architecture
   * Inference Stack: vLLM running on Cloud Run (gen2) with 16GiB RAM and 4 vCPUs. It uses GCS FUSE to mount model weights directly from Cloud
     Storage.
   * Agent Logic: Built with FastMCP, integrating google-cloud-logging for telemetry and the Vertex AI SDK (ADK) for model registry
     management.
   * Automation: A comprehensive Makefile handles environment setup, GPU deployment, and interactive model querying.

  The system is optimized for Private AI Operations, ensuring that infrastructure logs are analyzed by a private model endpoint rather than
  being sent to a public API.
