01

Why Cx

Sharp, deliberate, honest. Built for real-time work where performance, memory behavior, and tooling cost all matter.
THE PROBLEM

Real-time code can't afford surprises: unclear ownership, invisible overhead, unpredictable slow paths. Cx is built to make behavior and cost easy to see.

THE BET

Cx is trying to make the hard parts explicit: memory, lifetime, and cost - without requiring a heavyweight workflow. It's early and will change, but the direction is deliberate.

02

Features

Implemented vs planned - kept separate on purpose.
NOW
Functions
Direct, predictable function model.
Structs & grouped enums
Data types designed for engines.
Unified numeric types (t8-t128)
One numeric family. Fewer keyword fights.
Ternary values
Resolved / Unknown / Absent - data is honest.
Pattern matching (when blocks)
Branching that reads like intent.
Error messages
Clear errors. No guesswork.
LATER / IN PROGRESS
Type inference
In progress - will stay readable and explicit-first.
Modules / imports
Later - no premature architecture locks.
Generics
Later - only when the design is worth the complexity.
VM / bytecode
Later - not the current goal.
C interop
Later - planned, but not promised early.
Standard library
Later - minimal core first.

# run
cargo run --release

# optional direct binary run
# Linux/macOS: ./target/release/Cx_0V
# Windows: .\target\release\Cx_0V.exe
